**Section 1 — Context & Purpose**
The user is an F1 fan seeking a personalized AI assistant to provide detailed information about F1 races, results, analytics, and the history of the sport. The assistant's domain is Formula 1, and the tools are needed to support the user's passion for the sport by delivering real-time updates, in-depth analysis, and historical insights. Real-world usage scenarios include:
* Providing real-time updates on race results, standings, and news from the F1 world
* Analyzing data to offer insights on driver performances, team strategies, and circuit characteristics
* Engaging in conversations about F1, discussing topics like rule changes, driver rivalries, and the history of the sport
The tools will integrate with the AI assistant to provide a comprehensive and engaging experience for the user.

**Section 2 — Required Tools**
The following tools are required:
* `f1_race_details`: Provides detailed information about F1 races, including results, analytics, and post-race analysis.
	+ Input parameters: `race_id` (int), `season` (int)
	+ Output format: JSON-compatible dict containing `race_result`, `driver_standings`, `constructor_standings`, and `post_race_analysis`
	+ Key logic: Utilize APIs from reputable F1 data providers (e.g., Ergast API, Formula 1 API) to fetch and process race data
	+ Error handling: Implement retry mechanism with exponential backoff for network calls, handle API rate limits and errors
* `f1_analytics`: Offers in-depth analysis of F1 data, including driver and team performance metrics, technical data, and strategic analysis.
	+ Input parameters: `driver_id` (int), `team_id` (int), `season` (int)
	+ Output format: JSON-compatible dict containing `driver_performance`, `team_performance`, `technical_data`, and `strategic_analysis`
	+ Key logic: Leverage data from F1 APIs and apply machine learning algorithms to generate insights
	+ Error handling: Handle missing data, invalid input parameters, and API errors
* `f1_history`: Provides historical context and insights about the sport, including its early days, legendary drivers and teams, and significant events.
	+ Input parameters: `era` (str), `driver_id` (int), `team_id` (int)
	+ Output format: JSON-compatible dict containing `historical_context`, `legendary_drivers`, `legendary_teams`, and `significant_events`
	+ Key logic: Utilize a combination of F1 APIs, Wikipedia API, and natural language processing to generate historical content
	+ Error handling: Handle missing data, invalid input parameters, and API errors

**Section 3 — Architecture Requirements**
Create modular, reusable functions with clear docstrings and type hints. Use lightweight, production-friendly libraries (e.g., `requests`, `pandas`) and avoid unnecessary frameworks or abstractions. Implement robust try/except error handling, retries with exponential backoff for network calls, and configurable timeouts. Optimize for low latency and low memory usage.

**Section 4 — Recommended Libraries & APIs**
Recommended libraries include:
* `requests` for making HTTP requests to F1 APIs
* `pandas` for data manipulation and analysis
* `numpy` for numerical computations
* `scikit-learn` for machine learning tasks
Recommended APIs include:
* Ergast API for F1 data
* Formula 1 API for official F1 data
* Wikipedia API for historical context

**Section 5 — Quality Standards**
Ensure all functions are independently testable, inputs are validated before processing, and meaningful error messages are returned on failure. Use environment variables for API keys and secrets, and follow PEP 8 guidelines for code quality.
