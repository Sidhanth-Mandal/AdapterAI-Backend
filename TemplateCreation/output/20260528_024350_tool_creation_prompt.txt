**Section 1 — Context & Purpose**

- **User Goal:** Deliver a personalized Formula 1 (F1) assistant that supplies up‑to‑date race results, practice & qualifying times, and deep historical information. The assistant must also be able to compare drivers/teams, generate concise weekend summaries, and produce simple visualizations (lap‑time trends, tyre‑usage, points bar charts).  
- **Domain:** Motorsports – specifically the FIA Formula 1 World Championship. Real‑time data is required for current season events, while extensive historical data (seasons back to 1950) is needed for analysis and storytelling.  
- **Usage Scenarios:**  
  1. User asks “What were the qualifying times for the Monaco Grand Prix this weekend?” → tool fetches live/near‑live qualifying data.  
  2. User requests “Show me a table comparing Hamilton vs Verstappen lap times across the 2023 season.” → tool retrieves historical lap‑time data, builds a comparison table, and optionally returns a line chart.  
  3. User wants “Give me a short recap of the 2016 Singapore race weekend.” → tool assembles practice, qualifying, and race results into a bullet‑point narrative.  
  4. User asks for “Historical win‑rate of Ferrari at Monza.” → tool queries historical stats and returns a JSON snippet.  
- **Integration:** Each tool will be a pure Python function callable by the AI assistant via the OpenAI function‑calling interface. The assistant decides when to invoke a tool, passes the required arguments, and then formats the returned JSON for the user according to the tone guidelines.

**Section 3 — Architecture Requirements**

1. **Modularity:** Each tool must be a single, reusable function in its own file or module. No monolithic scripts; helper utilities (e.g., `api_get`, `retry_decorator`) may be shared.  
2. **Type Hints:** All signatures must use Python 3.9+ type hints, including `Literal` where appropriate.  
3. **Docstrings:** Every function (public and helper) must have a concise docstring describing purpose, parameters, return value, and raised exceptions.  
4. **Structured Outputs:** Functions return only JSON‑compatible Python objects (dicts, lists, primitives). No raw `print` statements.  
5. **Error Handling:** Wrap all external calls in `try/except`. Return a dict with `"status": "error"` and a human‑readable `"message"` field. Do not raise uncaught exceptions.  
6. **Retries:** Implement a reusable `@retry` decorator with exponential backoff (initial delay 1 s, factor 2, jitter ±0.1 s, max 3‑4 attempts). Apply to any network‑bound function.  
7. **Timeouts:** Use `httpx` (or `requests` if unavoidable) with explicit timeout parameters (`timeout=5` s for live calls, `timeout=8` s for historical).  
8. **Libraries:** Only lightweight, production‑grade libraries: `httpx`, `requests`, `pandas`, `matplotlib`, `beautifulsoup4` (for optional Wikipedia scraping), `functools`, `typing`. No heavy ML frameworks.  
9. **Performance:** Avoid loading entire season datasets into memory unless required; stream where possible. Cache immutable historical queries with `functools.lru_cache(maxsize=128)` or a simple file‑based JSON cache for Ergast responses (TTL 24 h).  
10. **Security:** No hard‑coded API keys. If a paid live‑timing API is used, read the key from `os.getenv("F1_LIVE_API_KEY")`.  

**Section 4 — Recommended Libraries & APIs**

- **HTTP Clients:** `httpx` (async optional, but sync is fine) for all REST calls.  
- **Data Manipulation:** `pandas` for table construction, sorting, merging.  
- **Visualization:** `matplotlib` (headless backend `Agg`) → PNG → base64.  
- **HTML Parsing (optional):** `beautifulsoup4` for extracting narrative snippets from Wikipedia pages.  
- **APIs:**  
  - **Ergast Developer API** – `http://ergast.com/api/f1/{year}/{...}.json` (free, no auth).  
  - **Official F1 Live‑Timing API** – placeholder endpoint; if unavailable, use a reputable third‑party like `https://api.formula1.com/v1/...` (requires `F1_LIVE_API_KEY`).  
  - **Wikipedia REST API** – `https://en.wikipedia.org/api/rest_v1/page/summary/{title}` for concise history paragraphs.  
- **Data Formats:** JSON for all API responses; CSV/Excel optional if user uploads personal logs (future extension).  
- **Caching:** `functools.lru_cache` for function‑level memoization; optional file‑based cache using `sqlite3` or simple JSON files for Ergast responses (TTL 24 h).  

**Section 5 — Quality Standards**

- **Testability:** Each function must be independently unit‑testable; avoid global state. Provide clear input‑output contracts.  
- **Input Validation:** Use `pydantic`‑style checks (or manual `isinstance`/`Literal` validation) to reject malformed arguments before any network call.  
- **Error Messages:** Return concise, user‑friendly messages (e.g., “Could not retrieve qualifying data for Monaco 2024 – the session may not have started yet.”).  
- **No Secrets:** All credentials read from environment variables; raise a clear error if missing.  
- **PEP 8 Compliance:** Follow standard naming, line length ≤ 100, import ordering. Include `# noqa: E501` only where unavoidable.  
- **Documentation:** In addition to docstrings, generate a short `README.md` snippet describing each tool, its parameters, and example calls (for downstream developers).
